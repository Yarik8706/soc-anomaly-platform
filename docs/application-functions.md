# Функциональные возможности SOC Anomaly Platform

## 1. Назначение приложения

**SOC Anomaly Platform** (в интерфейсе — **SOC Lens**) — веб-приложение для автоматизации части работы центра мониторинга информационной безопасности (SOC). Платформа принимает журналы событий SIEM и NGFW, проверяет и нормализует их, формирует поведенческие признаки пользователей и хостов, выполняет ML-поиск аномалий, помогает аналитику расследовать найденные сигналы и формирует SOC-отчёты.

Приложение объединяет в одном интерфейсе полный рабочий цикл:

1. вход пользователя в защищённую область;
2. загрузка журналов событий;
3. проверка структуры и нормализация данных;
4. запуск анализа в фоновом режиме;
5. построение признаков и расчёт anomaly score;
6. просмотр и фильтрация найденных аномалий;
7. изучение причин и контекста аномалии;
8. фиксация решения аналитика;
9. оценка прокси-метрик качества;
10. создание и экспорт SOC-отчёта;
11. аудит значимых действий пользователей.

## 2. Категории пользователей и права доступа

В приложении предусмотрено три роли.

| Возможность | Admin | Analyst | Viewer |
| --- | :---: | :---: | :---: |
| Вход в приложение и просмотр общей информации | Да | Да | Да |
| Просмотр загрузок, запусков, аномалий, отчётов и метрик | Да | Да | Да |
| Загрузка, проверка и нормализация файлов | Да | Да | Нет |
| Создание и повторный запуск анализа | Да | Да | Нет |
| Изменение статуса аномалии и добавление комментария | Да | Да | Нет |
| Формирование SOC-отчёта | Да | Да | Нет |
| Скачивание готовых отчётов | Да | Да | Да |
| Управление пользователями | Да | Нет | Нет |
| Просмотр журнала аудита | Да | Нет | Нет |

Отключённая учётная запись не может войти в систему. Меню администратора скрывается для остальных ролей, а сервер дополнительно проверяет права при каждом запросе.

## 3. Авторизация и пользовательская сессия

Приложение позволяет:

- войти по email и паролю;
- получить сведения о текущем пользователе и его роли;
- автоматически перенаправить неавторизованного пользователя на страницу входа;
- вернуться после входа к ранее запрошенной безопасной странице;
- выйти из системы;
- автоматически завершить локальную сессию при истечении или недействительности токена.

Backend выдаёт короткоживущий JWT-токен. Frontend сохраняет его в `HttpOnly` cookie через same-origin BFF-прокси, поэтому клиентский JavaScript не получает прямого доступа к токену. Refresh-токены и серверный список отозванных токенов в текущей версии не используются.

## 4. Обзорный экран

Раздел **«Обзор»** служит стартовой точкой приложения. Он:

- объясняет назначение платформы;
- показывает последовательность первого рабочего сценария: загрузка → нормализация → анализ;
- содержит места для ключевых показателей: числа новых аномалий, запусков анализа, объёма загруженных данных и среднего времени обработки.

В текущей реализации значения показателей на обзорном экране являются заглушками и не рассчитываются по данным backend. Рабочая статистика доступна в специализированных разделах «Аномалии», «Запуски» и «Метрики».

## 5. Загрузка исходных журналов

Раздел **«Загрузки»** позволяет подготовить входные данные для ML-анализа.

### 5.1. Добавление файлов

Пользователь с правом изменения данных может:

- выбрать файлы через системный диалог;
- перетащить их в область загрузки;
- сформировать очередь из нескольких файлов;
- удалить файл из очереди до отправки;
- загрузить один файл или пакет файлов;
- наблюдать прогресс и состояние загрузки каждого элемента;
- просматривать историю ранее загруженных файлов.

Ограничения загрузки:

- поддерживаются расширения `.csv`, `.tsv` и `.txt`;
- пустые файлы отклоняются;
- максимальный размер одного файла — **200 МиБ**;
- за один пакет можно отправить от 1 до 10 файлов;
- повторно выбранные одинаковые файлы не дублируются в клиентской очереди.

На сервере исходное имя сохраняется как метаданные, но физическое имя файла заменяется случайным UUID. Это исключает использование пользовательского имени как произвольного пути в файловой системе.

### 5.2. История и карточка файла

Для каждой загрузки приложение хранит и показывает:

- идентификатор;
- исходное имя;
- MIME-тип;
- размер;
- пользователя, выполнившего загрузку;
- дату создания;
- текущее состояние;
- дату проверки;
- дату нормализации;
- результат проверки структуры;
- список созданных нормализованных артефактов.

Основные состояния файла: `pending`, `validated`, `invalid`, `normalized`, `failed`.

## 6. Проверка структуры логов

Для загруженного файла можно отдельно запустить проверку. Платформа автоматически:

- определяет поддерживаемую кодировку: UTF-8 с BOM, UTF-8, CP1251; при наличии нулевых байтов также пробует UTF-16;
- определяет разделитель: табуляция, запятая, точка с запятой или вертикальная черта;
- читает заголовок и список колонок;
- проверяет наличие минимум двух колонок;
- выявляет пустые и повторяющиеся имена колонок;
- ищет временную колонку по известным именам либо по словам `time`, `timestamp`, `date`;
- проверяет, что файл содержит строки данных;
- на выборке до 1000 строк выявляет строки с некорректным количеством полей;
- возвращает понятный список найденных ошибок.

В карточке файла отображаются кодировка, разделитель, число проверенных строк, обнаруженные колонки и результат проверки. Файл с невалидной структурой нельзя нормализовать из интерфейса.

## 7. Нормализация журналов

Нормализация преобразует разнородные события в данные, пригодные для анализа. Во время обработки приложение:

- повторно проверяет структуру исходного файла;
- приводит временные значения к формату `YYYY-MM-DD HH:MM:SS`;
- добавляет отдельное поле `Date`;
- поддерживает несколько распространённых форматов даты и времени, включая ISO 8601;
- пропускает повреждённые строки и строки без распознаваемого времени;
- удаляет служебные поля `ClusterID`, `ClusterName`, `TenantID`, `TenantName`;
- анонимизирует значения `SourceUserName` и `DestinationUserName` в псевдонимы вида `user001`;
- игнорирует пустые, служебные и машинные учётные записи при построении пользовательского соответствия;
- определяет, относится ли источник к пользователю или хосту;
- распознаёт трафик Palo Alto Networks PAN-OS и отделяет его от событий SIEM;
- разделяет результат на отдельные CSV-файлы по дате и источнику `SIEM`/`PAN`;
- создаёт файл соответствия анонимизированных пользователей;
- считает число обработанных и пропущенных строк.

В интерфейсе результат представлен списком артефактов с именем файла, датой, типом источника и количеством строк. Внутренние пути показываются только как имена; прямого скачивания исходных и нормализованных файлов через текущий UI нет.

## 8. Создание и выполнение анализа

Раздел **«Запуски»** управляет ML-анализом.

### 8.1. Параметры запуска

При создании анализа пользователь может:

- выбрать один или несколько ранее нормализованных файлов;
- выбрать период анализа:
  - день;
  - неделя;
  - месяц;
  - произвольный диапазон;
  - все данные;
- указать опорную дату для дня, недели или месяца;
- указать начало и конец произвольного диапазона;
- задать число деревьев Isolation Forest от 10 до 1000;
- задать число сохраняемых результатов Top-N от 1 до 500.

Frontend проверяет обязательные даты, корректность диапазона и границы числовых параметров до отправки запроса. Backend дополнительно проверяет согласованность периода и дат.

### 8.2. Фоновое выполнение

Анализ ставится в очередь Redis и выполняется отдельным RQ worker, поэтому длительный ML-процесс не блокирует HTTP-запрос. Для запуска сохраняются:

- состояние;
- параметры и выбранные файлы;
- идентификатор фоновой задачи;
- текущий этап;
- число попыток;
- время создания, начала и завершения;
- созданные артефакты;
- безопасное сообщение об ошибке.

Основные состояния запуска: `pending`, `queued`, `running`, `completed`, `failed`.

Пользователь может просматривать историю запусков, фильтровать её по состоянию и периоду, открывать карточку запуска и наблюдать прогресс. Пока задача активна, карточка автоматически обновляется примерно раз в три секунды.

### 8.3. Этапы pipeline

В карточке отображаются этапы:

1. **Import** — получение нормализованных входных артефактов.
2. **Features** — построение признаков пользователей и хостов.
3. **Scoring** — обучение моделей и расчёт оценок аномальности.
4. **Explain** — формирование объяснений, контекста и сохранение аномалий.
5. **Reports** — зарезервированный этап; в текущей версии автоматическое создание отчёта внутри анализа не выполняется, отчёт запускается отдельно пользователем.

Для каждого этапа сохраняются состояние, время начала и окончания, а также ошибка. Из карточки доступны переходы к исходным загрузкам, аномалиям, метрикам и отчётам конкретного запуска.

### 8.4. Повторный запуск

Завершённый или неуспешный анализ можно поставить в очередь повторно. При этом:

- используется та же сущность запуска и те же параметры;
- счётчик попыток увеличивается;
- состояния этапов сбрасываются;
- предыдущая ошибка очищается;
- результаты аномалий этого запуска пересоздаются во время pipeline.

Отмена активной задачи, изменение параметров существующего запуска и удаление запуска в текущей версии не реализованы.

## 9. Построение признаков

Платформа объединяет выбранные нормализованные CSV и строит дневные профили двух типов сущностей:

- **пользователи** — по `SourceUserName` и `DestinationUserName`;
- **хосты** — по полям `Host`, `Hostname`, `SourceHostName`, `DestinationHostName`, `SourceAddress`.

Для каждой пары «сущность + дата» рассчитываются, в частности:

- общее число событий;
- количество уникальных часов активности;
- доля ночной активности с 00:00 до 05:59;
- доля активности в рабочее время с 09:00 до 17:59;
- количество уникальных адресов назначения;
- количество уникальных исходных процессов;
- количество уникальных классов событий;
- количество уникальных категорий;
- количество уникальных имён событий.

Признаки сохраняются как отдельные артефакты для пользователей и хостов.

## 10. ML-поиск аномалий

Аномальность рассчитывается отдельно для пользователей и хостов. Платформа:

- выбирает целевую дату из параметров запуска либо последнюю доступную дату;
- использует остальные даты как историческую обучающую выборку;
- преобразует признаки в числовую матрицу;
- применяет устойчивое масштабирование `RobustScaler`;
- обучает `IsolationForest`;
- обучает `LocalOutlierFactor` в режиме novelty detection, если исторических объектов достаточно;
- нормализует оценки обеих моделей;
- усредняет их в итоговый `score` от 0 до 1 внутри рассматриваемой выборки;
- ранжирует сущности по убыванию аномальности;
- сохраняет заданное число Top-N результатов.

По позиции в рейтинге сигнал получает критичность:

- `critical` — верхние 5%;
- `high` — следующие позиции до 20%;
- `medium` — до 50%;
- `low` — остальные сохранённые результаты.

Итоговый score является сравнительной оценкой аномальности, а не вероятностью атаки или подтверждённого инцидента.

## 11. Очередь и поиск аномалий

Раздел **«Аномалии»** предоставляет рабочее пространство аналитика. Он позволяет:

- просматривать общее число найденных сигналов;
- видеть счётчики критических сигналов, сигналов в расследовании и подтверждённых инцидентов;
- фильтровать данные по ID запуска;
- задавать диапазон дат;
- выбирать тип сущности: пользователь или хост;
- фильтровать по критичности;
- фильтровать по статусу расследования;
- менять число элементов на странице;
- переходить между страницами;
- менять порядок элементов текущей страницы;
- открывать подробную карточку сигнала;
- сохранять фильтры в URL и возвращаться из карточки к той же выборке.

Сигнал содержит сущность, дату, тип, итоговый score, место в рейтинге, критичность, краткое описание, статус workflow, контекст и идентификатор запуска.

## 12. Объяснимость и контекст аномалии

В карточке аномалии отображаются причины, по которым объект отличается от исторического поведения. Для каждого из наиболее значимых признаков показываются:

- имя признака;
- значение в анализируемый день;
- исторический baseline;
- нормированный вклад отклонения;
- визуальная шкала величины вклада.

Baseline строится по истории той же сущности. Если её недостаточно, используется история остальных сущностей. Для устойчивой оценки отклонения применяется медиана и медианное абсолютное отклонение, а при невозможности — стандартное отклонение.

Дополнительный контекст собирается из исходных нормализованных событий и может включать:

- количество связанных событий;
- IP-адреса;
- процессы;
- имена, категории и классы событий;
- связанных пользователей;
- часы активности;
- отдельные оценки Isolation Forest и LOF.

Объяснение помогает приоритизировать расследование, но не заменяет экспертное решение SOC-аналитика.

## 13. Workflow расследования

Для каждой аномалии предусмотрены статусы:

- `new` — новый сигнал;
- `investigating` — выполняется проверка;
- `incident` — подтверждённый инцидент;
- `false_positive` — ложное срабатывание;
- `closed` — работа завершена.

Admin и analyst могут изменить статус и добавить комментарий. Комментарий обязателен при переводе сигнала в `incident`, `false_positive` или `closed`. Каждое изменение создаёт запись истории со старым и новым статусом, комментарием, временем и идентификатором исполнителя.

Изменение статуса также попадает в общий журнал аудита. Подтверждение инцидента записывается как критическое событие.

## 14. SOC-отчёты

Раздел **«Отчёты»** позволяет:

- выбрать завершённый запуск анализа;
- поставить формирование отчёта в фоновую очередь;
- видеть состояния `queued`, `running`, `completed`, `failed`;
- автоматически получать обновление статуса активного отчёта;
- открыть готовый отчёт в интерфейсе;
- просмотреть его Markdown-содержимое;
- скачать результат в Markdown;
- скачать результат в PDF;
- просмотреть историю созданных отчётов;
- отфильтровать отчёты по запуску через параметр URL.

Отчёт содержит:

- идентификатор и период запуска;
- дату формирования;
- сводку количества аномалий по уровням критичности;
- таблицу аномалий с рангом, типом сущности, именем, датой, критичностью, score и описанием.

В PDF включается сводка и до 100 аномалий. Скачивание отчёта фиксируется в журнале аудита. Одновременно для одного запуска нельзя создавать несколько активных отчётов.

## 15. Метрики качества и стабильности

Раздел **«Метрики»** работает с завершёнными запусками и показывает прокси-метрики, рассчитанные по результатам поиска аномалий:

- гистограммы распределения anomaly score отдельно для пользователей и хостов;
- стабильность Top-K относительно предыдущего завершённого запуска;
- `Jaccard@K` — сходство множеств наиболее аномальных сущностей;
- `Overlap@K` — долю пересечения Top-K;
- `Spearman@K` — согласованность порядка общих сущностей;
- список признаков, наиболее часто встречающихся в объяснениях аномалий;
- дату расчёта метрик и идентификатор сравниваемого запуска.

Если предыдущего запуска или достаточного числа общих объектов нет, показатели стабильности отображаются как недоступные. Рассчитанные метрики кэшируются в базе данных.

Эти показатели являются прокси-метриками согласованности результатов, а не полноценной оценкой precision/recall: в приложении нет размеченного эталонного набора атак.

## 16. Управление пользователями

Раздел **«Пользователи»** доступен только администратору. Он позволяет:

- просматривать список учётных записей;
- видеть email, UUID, роль, активность и дату создания;
- создавать пользователя с ролью admin, analyst или viewer;
- назначать и изменять роль;
- устанавливать новый пароль;
- включать и отключать учётную запись.

Пароль новой учётной записи и новый пароль существующей должны содержать не менее 12 символов. Пароли хранятся только в виде адаптивного хеша. Email нормализуется к нижнему регистру; создание дубликата запрещено.

## 17. Журнал аудита

Раздел **«Аудит»** доступен только администратору и содержит значимые действия:

- загрузку файла;
- запуск анализа;
- повторный запуск анализа;
- изменение статуса аномалии;
- создание отчёта;
- экспорт отчёта;
- создание пользователя;
- изменение пользователя.

Для события сохраняются время, действие, тип и ID объекта, пользователь, важность и дополнительные JSON-данные. Журнал можно фильтровать по точному имени действия, важности и типу объекта, просматривать постранично и открывать полные детали в модальном окне.

Под «неизменяемым» журналом в пользовательском интерфейсе понимается отсутствие операций редактирования и удаления через API/UI. Криптографическая неизменяемость или внешнее WORM-хранилище в текущей реализации не предусмотрены.

## 18. Эксплуатационные возможности

Приложение включает:

- публичную проверку работоспособности API;
- отдельную проверку подключения к PostgreSQL;
- документированный OpenAPI/Swagger UI;
- хранение метаданных в PostgreSQL;
- хранение исходных и производных файлов в файловом хранилище;
- Redis и RQ для фоновых задач;
- отдельный worker для анализа и отчётов;
- миграции Alembic;
- идемпотентное создание первого администратора;
- Docker Compose для совместного запуска frontend, backend, PostgreSQL, Redis, worker и init-процессов;
- автоматическое обновление интерфейса для активных запусков и отчётов;
- обработку типовых состояний загрузки, пустых данных и ошибок;
- адаптивную боковую навигацию для узких экранов.

Артефакты каждого запуска изолируются в каталоге `data/runs/<run-id>/`. Сервер проверяет, что скачиваемый отчёт находится внутри каталога соответствующего запуска, и не принимает произвольный файловый путь от клиента.

## 19. Типовой пользовательский сценарий

1. Администратор создаёт учётные записи аналитиков и наблюдателей.
2. Аналитик входит в систему и открывает раздел «Загрузки».
3. Аналитик добавляет CSV/TSV/TXT-журналы SIEM или NGFW.
4. Платформа проверяет кодировку, разделитель, заголовок и временную колонку.
5. Аналитик запускает нормализацию и проверяет список дневных SIEM/PAN-артефактов.
6. В разделе «Запуски» аналитик выбирает нормализованные файлы, период и параметры ML.
7. Worker строит признаки, выполняет скоринг и сохраняет Top-N аномалий.
8. Аналитик открывает очередь аномалий и приоритизирует сигналы по критичности.
9. В карточке сигнала аналитик изучает отклонённые признаки, baseline и контекст событий.
10. Аналитик переводит сигнал в расследование, инцидент, ложное срабатывание или закрытое состояние и фиксирует обоснование.
11. В разделе «Метрики» аналитик оценивает распределение score и устойчивость Top-K.
12. После завершения анализа аналитик создаёт SOC-отчёт и экспортирует Markdown или PDF.
13. Администратор при необходимости проверяет значимые действия в журнале аудита.

## 20. Границы текущей реализации

Чтобы корректно интерпретировать возможности приложения, важно учитывать следующие ограничения:

- показатели обзорного экрана пока не подключены к реальным данным;
- приложение обнаруживает статистические аномалии, но не подтверждает факт атаки автоматически;
- отсутствует размеченная выборка для расчёта precision, recall и других supervised-метрик;
- формирование отчёта запускается отдельно и не является автоматическим этапом основного pipeline;
- нельзя отменять активные фоновые задачи;
- нельзя удалить загрузку, запуск, аномалию, отчёт или запись аудита через UI/API;
- нельзя редактировать параметры уже созданного запуска;
- исходные и нормализованные файлы нельзя скачать через текущий интерфейс;
- отсутствуют refresh-токены, SSO, MFA и централизованный отзыв JWT;
- нет уведомлений по email, Slack или другим внешним каналам;
- нет интеграции с внешней SIEM/SOAR для автоматического получения логов или создания инцидентов;
- журнал аудита не имеет криптографической защиты от изменения на уровне хранилища;
- диапазон, неделя и месяц задают контекст запуска, но текущий scoring выбирает одну целевую дату: опорную дату либо конец диапазона; это следует учитывать при интерпретации результатов.

## 21. Итог

SOC Anomaly Platform реализует базовый end-to-end процесс поведенческого анализа событий безопасности: безопасный приём данных, нормализацию, фоновый ML-скоринг пользователей и хостов, объяснение результатов, workflow расследования, прокси-метрики, отчётность, ролевое управление и аудит. Платформа подходит как основа внутреннего инструмента SOC и как расширяемый MVP для последующей интеграции с корпоративными источниками данных и процессами реагирования.
